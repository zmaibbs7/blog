量化交易平台 — 软件架构设计文档 v3.0（最终版）

基于 v2.0 + 外部深度评审修订 核心新增：时间模型、资金会计、订单状态机、调度监控 每个模块标注实施阶段（P1/P2/P3），避免过度工程化

设计纲领

[!IMPORTANT] 本文档的核心原则：每个模块都标注了实施阶段。

  • P1（Phase 1）：必须首先实现，系统才能运行
  • P2（Phase 2）：系统稳定运行后，解决可靠性和一致性
  • P3（Phase 3）：规模化、智能化扩展

Phase 1 的唯一目标：单机、单策略、单 Broker、稳定回测 + 稳定实盘

第二部分：核心子系统设计

以下是 v3.0 新增或重大修订的子系统，它们解决 v2.0 最薄弱的"系统架构"层面

2.1 ⭐ 时间系统 time/ — P1

v2.0 最大缺失。量化平台是"时间驱动系统"，没有统一时钟一切都是不稳定的。

time/
├── core/
│   ├── clock.py                 # IClock 接口
│   ├── calendar.py              # IExchangeCalendar 接口
│   └── types.py                 # Timestamp, TimeRange, TradingSession
│
├── clocks/
│   ├── simulated_clock.py       # 回测用：模拟时间，可加速/暂停
│   ├── realtime_clock.py        # 实盘用：真实墙钟
│   └── replay_clock.py          # 回放用：按历史时间戳推进
│
├── calendars/
│   ├── china_stock.py           # A股：09:30-11:30, 13:00-15:00
│   ├── china_futures.py         # 期货：含夜盘
│   ├── us_stock.py              # 美股
│   └── crypto.py                # 加密货币：7×24
│
└── utils.py                     # 时区转换、时间对齐

时间模型的五种时间：

┌─────────────────────────────────────────────────────────┐
│                   时间模型                                │
│                                                          │
│  Exchange Time ─── 交易所时间（北京/纽约/UTC）             │
│       ↓                                                  │
│  Event Time ────── 事件发生时间（行情时间戳）               │
│       ↓                                                  │
│  Processing Time ─ 策略处理时间（有延迟）                  │
│       ↓                                                  │
│  Execution Time ── 订单执行时间                           │
│                                                          │
│  回测中：以上全部由 SimulatedClock 统一模拟                │
│  实盘中：Exchange Time 来自交易所，其余来自 RealtimeClock  │
└─────────────────────────────────────────────────────────┘

接口定义：

IClock
├── now() → Timestamp                     # 当前时间
├── is_trading_time() → bool              # 是否在交易时段
├── sleep(duration)                       # 等待（回测=快进，实盘=真等）
└── schedule(time, callback)              # 定时触发

IExchangeCalendar
├── is_trading_day(date) → bool
├── get_trading_sessions(date) → list[TradingSession]
├── next_trading_day(date) → date
├── prev_trading_day(date) → date
└── get_trading_minutes(date) → list[Timestamp]

TradingSession
├── exchange: Exchange
├── open_time: time
├── close_time: time
├── pre_market: time | None               # 盘前（美股有）
├── post_market: time | None              # 盘后
└── timezone: str

为什么这是 P1：

没有时间模型 →
  回测看到未来数据（未来函数）
  多市场时间错位
  实盘 vs 回测时间不一致
  → 所有回测结果不可信

2.2 ⭐ 资金会计 accounting/ — P1

v2.0 的 Portfolio 太单薄。资金管理是量化系统"最容易出错且最难调试"的部分。

accounting/
├── core/
│   ├── account.py               # IAccount 接口
│   ├── position_book.py         # IPositionBook 接口
│   ├── ledger.py                # ILedger 接口（记账本）
│   └── types.py                 # CapitalSnapshot, PnLBreakdown
│
├── account.py                   # Account 实现
│                                #   总资产 = 可用资金 + 冻结资金 + 持仓市值
│
├── position_book.py             # PositionBook 实现
│                                #   按品种管理持仓，支持 FIFO/平均成本
│
├── ledger.py                    # Ledger 实现
│                                #   每笔资金变动的不可变记录
│
└── settlement.py                # 结算处理
                                 #   逐日盯市（期货）/ T+1 交收（股票）

Account 的资金模型：

Account 资金结构
─────────────────────────────────
  总权益 (equity)
    = 可用资金 (available)
    + 冻结资金 (frozen)         ← 未成交订单冻结的资金
    + 持仓市值 (market_value)   ← 所有 Position 的当前市值

  冻结资金 (frozen)
    = 待成交买单金额 + 预扣手续费

  已实现盈亏 (realized_pnl)
    = Σ 每笔平仓的盈亏

  未实现盈亏 (unrealized_pnl)
    = Σ 每个持仓的 (当前价 - 持仓均价) × 数量

为什么从 Portfolio 独立出来：

Portfolio（v2.0）= 交易概念，偏用户视角
Accounting（v3.0）= 精确簿记，偏系统视角

区别：
  Portfolio: "我持有100股平安银行"
  Accounting: "100股平安银行，成本12.35，冻结资金1235，
              手续费预扣6.18，保证金占用0（股票），
              逐笔记录在 Ledger"

2.3 ⭐ 订单管理系统 oms/ — P1

v2.0 的 IBroker.submit_order() 过于理想化。真实订单有复杂的生命周期。

oms/
├── core/
│   ├── order_manager.py         # IOrderManager 接口
│   ├── state_machine.py         # OrderStateMachine 定义
│   └── types.py                 # ExecutionReport
│
├── state_machine.py             # 订单状态机实现
├── order_manager.py             # OrderManager 实现
├── order_book.py                # 活跃订单簿
└── idempotency.py               # 幂等性保证（防重复下单）

订单状态机：

                    ┌──────────┐
                    │   NEW    │ ← 策略创建
                    └────┬─────┘
                         │ submit
                    ┌────▼─────┐
              ┌─────│ PENDING  │─────┐
              │     └────┬─────┘     │
              │          │           │
         cancel     partial fill   reject
              │          │           │
        ┌─────▼────┐ ┌───▼──────┐ ┌─▼────────┐
        │ CANCEL   │ │ PARTIAL  │ │ REJECTED │
        │ PENDING  │ │ FILLED   │ └──────────┘
        └─────┬────┘ └───┬──────┘
              │           │
         confirmed    fill remaining
              │           │
        ┌─────▼────┐ ┌───▼──────┐
        │CANCELLED │ │  FILLED  │
        └──────────┘ └──────────┘

        ┌──────────┐
        │ EXPIRED  │ ← 超时自动取消
        └──────────┘

合法状态转换：
  NEW → PENDING                     提交到 Broker
  PENDING → PARTIAL_FILLED          部分成交
  PENDING → FILLED                  全部成交
  PENDING → REJECTED                被拒绝（资金不足/风控/交易所）
  PENDING → CANCEL_PENDING          请求撤单
  PARTIAL_FILLED → FILLED           剩余全部成交
  PARTIAL_FILLED → CANCEL_PENDING   撤销剩余
  CANCEL_PENDING → CANCELLED        撤单确认
  PENDING → EXPIRED                 超时

非法转换（系统必须拒绝）：
  FILLED → 任何状态               ❌
  CANCELLED → 任何状态            ❌
  REJECTED → 任何状态             ❌

ExecutionReport：

ExecutionReport                       # 每次状态变更产生一条
├── order_id: str
├── exec_type: ExecType              # NEW / PARTIAL / FILL / CANCEL / REJECT
├── order_status: OrderStatus        # 变更后的状态
├── filled_qty: float                # 本次成交数量
├── filled_price: float              # 本次成交价格
├── cumulative_qty: float            # 累计成交数量
├── remaining_qty: float             # 剩余未成交
├── commission: float                # 本次手续费
├── text: str                        # 附加信息（拒绝原因等）
└── timestamp: Timestamp

事件总线契约（v3.0 新增）：

IEventBus 契约
─────────────────────────────────────────────
  顺序保证：
    同一 publisher 发出的事件，subscriber 按发出顺序接收
    不同 publisher 的事件，不保证跨 publisher 顺序

  投递保证：
    P1：At-most-once（丢失可接受，简单优先）
    P2：At-least-once（重复可接受，通过幂等性去重）

  同步 vs 异步：
    SyncEventBus：   publish() 阻塞直到所有 subscriber 处理完
                     适合回测（确定性、可重复）
    AsyncEventBus：  publish() 立即返回，subscriber 异步处理
                     适合实盘（低延迟）

  背压策略（P2）：
    当 subscriber 处理速度 < publisher 发送速度：
    - 回测：不可能发生（同步）
    - 实盘：丢弃旧事件 / 合并同类事件 / 报警

IBroker 回调模型（v3.0 修正）：

IBroker
├── submit_order(order) → order_id

BacktestBroker 实现：
  submit_order() → 内部同步撮合 → 构造 ExecutionReport → 立即回调
  → 对策略完全透明，和实盘一样通过回调通知

LiveBroker 实现：
  submit_order() → 发送到交易所 → 等待异步回报 → 收到后构造 ExecutionReport → 回调
