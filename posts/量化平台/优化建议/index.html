量化交易平台 v3.0 架构评估

总体印象

这份文档是一份相当成熟的架构设计，v3.0 相比 v2.0 有显著改进，特别是时间模型、OMS状态机、资金会计的引入。但仍存在若干系统性问题值得深入讨论。

优点

分层设计清晰。用户入口层 → 平台核心层 → 数据层的三层分离符合量化平台的最佳实践，核心概念（时钟、撮合、账户）有明确的抽象接口。

阶段化落地务实。P1/P2/P3 的拆分避免了过度工程化，Phase 1 聚焦"单机、单策略、稳定跑通"是正确的优先级判断。

OMS 状态机设计正确。订单状态的合法/非法转换图完备，ExecutionReport 的引入统一了状态变更的通知模型，这是 v2.0 最大的进步之一。

VectorStrategy “弱状态"修正。将纯无状态改为携带计算状态 generate_signals(bars, state) → (SignalArray, state) 是务实的，避免了在线特征无法缓存的问题。

问题与优化建议

🔴 严重问题（会导致系统在实盘中出错）

问题一：StrategyContext 持有 IBroker 的安全隐患

文档中 StrategyContext.broker: IBroker 允许策略直接调用 broker.submit_order()，绕过风控链。

# 危险：策略可以绕过风控直接下单
class MyStrategy(EventStrategy):
    def on_bar(self, bar, ctx):
        ctx.broker.submit_order(Order(...))  # 跳过了 risk_chain.check()
        ctx.buy(...)                          # 走正常风控路径

建议：Context 只暴露 buy/sell/close 便捷方法，内部由引擎统一走风控链，彻底移除 context.broker 的直接引用，或者改成只读的 context.broker_status（仅查询，不可操作）。

问题二：时间模型缺少"数据截断"的强制机制

文档声明 DataView 会"自动截断到 now()"，但没有定义如何强制实施。

# 策略可能轻易拿到未来数据
class MyStrategy(EventStrategy):
    def on_bar(self, bar, ctx):
        # history() 内部是否真的被截断？
        # 如果策略直接访问 DataManager 而非 ctx.history()，怎么办？
        future_data = self.data_manager.get_bars("SH.600000", end="2025-01-01")

未来函数（Look-ahead bias）是回测失效的最常见原因，文档只在 StrategyContext.history() 层面提到截断，但没有在架构层面做硬性隔离。

建议：引入 DataView 作为数据访问的唯一代理，在引擎构建 Context 时传入 current_timestamp，DataView 内部拦截所有访问并做时间校验。任何绕过 DataView 直接访问 DataManager 的路径应在 P1 设计时就封闭。

class DataView:
    def __init__(self, manager: DataManager, as_of: Timestamp):
        self._manager = manager
        self._as_of = as_of  # 严格的时间截止点

    def get_bars(self, symbol, n) -> BarData:
        # 强制过滤，物理上不可能拿到未来数据
        return self._manager.get_bars(symbol, end=self._as_of, n=n)

问题三：Account 并发安全未定义

实盘中，broker.on_execution_report() 是异步回调，可能在独立线程中触发 account.on_execution()，而主线程同时在读取 account.equity() 供策略决策。

主线程：strategy.on_bar() → ctx.equity() → account.equity()  ← 读
回调线程：                  exec_report → account.on_execution() ← 写

文档完全没有提到 Account 的线程安全保证。

建议：明确 Account 的并发模型。推荐方案是单线程账户更新——所有 ExecutionReport 通过 AsyncEventBus 排队到主线程处理，Account 始终在单线程内操作。文档中 AsyncEventBus 的描述需要明确这一约束。

🟡 设计缺陷（影响健壮性和可维护性）

问题四：IBroker 接口的"查询"和"命令"混用

IBroker
├── submit_order()      # 命令
├── cancel_order()      # 命令
├── get_order()         # 查询
├── get_position()      # 查询
├── get_portfolio()     # 查询

将查询和命令混在一个接口里，导致测试时需要 mock 整个 Broker 才能测试账户同步逻辑。

建议：参考 CQRS 原则拆分为两个接口。

class IBrokerCommands:
    def submit_order(self, order) -> str: ...
    def cancel_order(self, order_id) -> bool: ...

class IBrokerQueries:
    def get_order(self, order_id) -> Order: ...
    def get_position(self, symbol) -> Position: ...

问题五：RiskChain 的返回值设计不完整

文档描述了 risk_chain.check(order)，但没有定义返回值结构。风控拒绝时是抛异常还是返回结果对象？这直接影响策略能否感知风控决策。

建议：定义明确的 RiskCheckResult：

@dataclass
class RiskCheckResult:
    passed: bool
    rule_id: str | None      # 哪条规则触发
    reason: str | None       # 拒绝原因
    modified_order: Order | None  # 风控修改了订单（如限仓位）

# RiskChain 返回 RiskCheckResult，不抛异常
# 引擎根据 result.passed 决定是否继续提交

问题六：Ledger 与 Account 的一致性保证缺失

文档引入了 Ledger 做"不可变记账”，但没有说明 Ledger 和 Account 状态如何保持一致。如果 Account 更新成功但 Ledger 写入失败，系统处于不一致状态。

建议：明确 Account 更新和 Ledger 写入是同一事务（即使是内存事务）。P2 引入 Persistence 后，需要保证 Ledger 持久化和 Account 快照的原子性。

问题七：Bar 内事件排序的启发式方法不可靠

文档描述 realistic_match 中"基于 open→high/low 距离判断高低哪个先到达"。这是一个启发式假设，在实际中并不可靠，且会导致回测结果不可重复（因为相同 OHLC 可能有不同路径）。

建议：提供明确的策略选项而不是一个模糊的启发式。

class BarIntraOrderPolicy(Enum):
    CONSERVATIVE = "high_then_low"   # 止损优先（保守）
    AGGRESSIVE   = "low_then_high"   # 止盈优先（激进）
    RANDOM       = "random"          # 蒙特卡洛随机化
    OPEN_ONLY    = "open_price"      # 只用开盘价，最简单

至少让用户可以选择，并明确每种选择的含义。

🟢 架构层面的改进建议

建议一：补充 tests/ 的测试策略

文档列出了 unit/integration/regression/benchmark 四类测试目录，但完全没有说明测试策略。对量化平台而言，最关键的是：

  • 确定性测试：相同配置、相同数据、相同随机种子 → 回测结果必须 bit-for-bit 相同
  • Broker Mock 规范：如何 mock ExecutionReport 的异步到来
  • 时间注入测试：SimulatedClock 需要有完整的单元测试覆盖所有边界（跨日、节假日、夜盘开始/结束）

这部分应该在 P1 规划中明确，而不是留空。

建议二：config/container.py 的 DI 方案需具体化

DI 容器是整个系统的"神经中枢"，但文档只提到"YAML 配置 + DI 容器"，没有说明用哪个 DI 框架（dependency-injector？自制？wireup？）以及如何处理不同阶段（回测 vs 实盘）的不同依赖绑定。

建议：明确两套 DI Profile。

# config.yaml
mode: backtest  # or live

backtest:
  clock: SimulatedClock
  event_bus: SyncEventBus
  broker: BacktestBroker

live:
  clock: RealtimeClock
  event_bus: AsyncEventBus
  broker: XTBroker

引擎启动时根据 mode 自动切换依赖，策略代码完全不感知。

建议三：补充 types/ 的不变性保证

文档将 types/ 定义为"Level 0 禁止修改"的核心数据结构，但没有说明这些数据结构是可变的还是不可变的。

推荐所有核心类型使用 frozen dataclass 或 NamedTuple：

@dataclass(frozen=True)  # 不可变
class ExecutionReport:
    order_id: str
    filled_qty: float
    ...
    # 禁止 exec_report.filled_qty = 100 这种危险修改

这对审计日志和事件溯源的正确性至关重要。

总结评分

维度 评分 说明
整体架构 ★★★★☆ 分层合理，抽象到位
时间模型 ★★★★★ v3.0 最大亮点，设计完备
OMS 状态机 ★★★★☆ 完整但缺少幂等性细节
资金会计 ★★★★☆ 引入 Ledger 是正确的，但一致性未定义
并发安全 ★★☆☆☆ 最大盲点，实盘必然踩坑
未来函数防护 ★★★☆☆ 架构层面未做硬隔离
风控设计 ★★★☆☆ 返回值和绕过风险未处理
测试策略 ★★☆☆☆ 几乎空白
落地可行性 ★★★★☆ P1/P2/P3 划分务实

核心建议优先级：先解决并发安全（实盘会出资金错误）→ 再解决未来函数硬隔离（回测结果不可信）→ 再完善风控链返回值 → 其余可在迭代中修正。
