Hugo + Obsidian 目录设计说明

目标

这套目录设计的目标是把 Obsidian 的笔记管理能力和 Hugo 的静态站点生成能力解耦开来：

  • Obsidian 负责管理整个笔记仓库
  • Hugo 只负责把博客相关的 Markdown 转成静态网页
  • GitHub Pages 只负责发布 Hugo 生成的网站

这样做的好处是，博客内容和其他私人笔记、项目笔记、日记不会混在一起，同时 Hugo 站点的职责也保持单一。

设计原则

目录设计遵循下面几条原则：

  • notes/ 是 Obsidian Vault 根目录，也是日常使用的主目录
  • blog/ 是公开博客内容源，只放准备发布的内容
  • site/ 是 Hugo 工程目录，只负责构建和发布
  • 非博客内容默认不进入 Hugo 发布链路
  • 第一版用相对路径软链接连接 blog/ 和 site/

推荐目录结构

notes/
  .obsidian/
  inbox/
  daily/
  knowledge/
  project/
  blog/
    content/
      posts/
      pages/
    assets/
      images/
  site/
    content/
      posts -> ../../blog/content/posts
      pages -> ../../blog/content/pages
    static/
      images -> ../../blog/assets/images
    .github/
      workflows/
        hugo.yml
    hugo.toml

各目录职责

notes/

这是 Obsidian 的 Vault 根目录。你在 Obsidian 里应该直接打开这个目录，而不是只打开 blog/ 或 site/。

blog/

这是博客内容源目录。所有准备公开发布的内容都应该放在这里。

建议约定如下：

  • blog/content/posts/：博客文章
  • blog/content/pages/：单页内容，例如 About、Links、Now
  • blog/assets/images/：博客图片资源

这个目录由 Obsidian 主要负责管理和编辑。

site/

这是 Hugo 站点目录。它的作用是：

  • 保存 Hugo 配置
  • 保存主题、模板和发布配置
  • 作为 GitHub Pages 的发布入口

你应该在 site/ 目录中运行 Hugo 命令，例如：

cd site
hugo server -D

为什么第一版使用软链接

第一版使用相对路径软链接，是因为它能在保留目录边界的前提下，避免手工复制文件。

这意味着：

  • 你在 blog/content/posts/ 中写文章
  • Hugo 从 site/content/posts 读取文章
  • 但 site/content/posts 实际上是一个指向 blog/content/posts/ 的软链接

这样既不需要重复保存内容，也不需要先写同步脚本，足够适合第一版落地。

软链接创建方式

先确保 site/ 已经初始化为一个 Hugo 站点，然后在 notes/ 根目录执行：

mkdir -p site/content site/static
ln -s ../../blog/content/posts site/content/posts
ln -s ../../blog/content/pages site/content/pages
ln -s ../../blog/assets/images site/static/images

创建完成后，site/ 中会出现三处链接关系：

  • site/content/posts -> ../../blog/content/posts
  • site/content/pages -> ../../blog/content/pages
  • site/static/images -> ../../blog/assets/images

为什么不把整个 blog/ 都直接链接进 site/

不建议把整个 blog/ 一次性链接为 site/content/，原因是边界会变模糊。

更细粒度的三个链接更容易维护，因为它明确了：

  • 哪些内容会进入 Hugo
  • 哪些内容属于文章
  • 哪些内容属于单页
  • 哪些内容属于静态资源

这样以后即使要替换为同步脚本，也只需要替换这几个入口。

使用规则

为了让这套结构长期稳定，建议固定以下规则：

  • 博客文章只写在 blog/content/posts/
  • About、Links、Now 等单页只写在 blog/content/pages/
  • 博客图片只放在 blog/assets/images/
  • site/content/ 和 site/static/ 不手工写文件
  • Hugo 命令只在 site/ 中执行
  • GitHub Pages 只发布 site/

推荐工作流

日常工作建议按下面的顺序进行：

  1. 在 Obsidian 中打开 notes/
  2. 在 blog/content/posts/ 中新建或编辑文章
  3. 如需插图，把图片放进 blog/assets/images/
  4. 进入 site/ 运行 hugo server -D
  5. 本地预览确认无误后，提交并推送 site/ 的发布配置以及 blog/ 中的内容
  6. 由 GitHub Actions 构建并发布到 GitHub Pages

这种设计的优点

  • 博客内容和私人笔记分离，发布边界清楚
  • Obsidian 仍然可以管理完整知识库
  • Hugo 工程职责单一，更容易维护
  • 不需要第一天就引入同步脚本
  • 如果将来需要改成同步脚本，迁移成本很低

后续可演进方向

如果后续发现软链接在 CI 或跨环境使用中不够稳定，可以把当前结构平滑迁移为“blog/ 作为内容源，脚本同步到 site/”的方案。

那时不需要改变内容组织方式，只需要把链接替换成同步步骤即可。
